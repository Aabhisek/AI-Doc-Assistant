# 4. Vector Database (pgvector)

## What Is a Vector?

A vector is an ordered list of numbers. In NLP, an **embedding** is a vector that represents the semantic meaning of a text.

Example (simplified — real vectors have 1024+ dimensions):
```
"Java developer"      → [0.82, 0.14, 0.67, 0.33, ...]
"Software engineer"   → [0.79, 0.18, 0.71, 0.28, ...]  ← close!
"Recipe for pasta"    → [0.11, 0.91, 0.03, 0.77, ...]  ← far away
```

The proximity in vector space corresponds to semantic similarity.

---

## What Is pgvector?

pgvector is an **open-source PostgreSQL extension** that adds:
- A `vector` column type
- Similarity operators (`<=>` for cosine, `<->` for L2, `<#>` for inner product)
- Indexes for fast approximate nearest-neighbor (ANN) search

The advantage over dedicated vector databases (Pinecone, Weaviate, Qdrant):
- No extra infrastructure
- Uses your existing PostgreSQL instance
- Full SQL available for hybrid searches (vector + relational filters)
- Simpler operational model for small/medium scale

## How Spring AI Uses pgvector

Spring AI creates a `vector_store` table automatically:

```sql
CREATE TABLE IF NOT EXISTS vector_store (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    content     TEXT,
    metadata    JSONB,
    embedding   vector(1024)
);

CREATE INDEX IF NOT EXISTS vector_store_embedding_idx
    ON vector_store USING hnsw (embedding vector_cosine_ops);
```

The `VectorStoreConfig` bean is created manually with `dimensions=1024` and `initializeSchema=true`, which auto-creates the extension and table on first startup.

The `metadata` column stores key-value pairs we attach to each chunk (e.g., `documentId`, `documentName`). Spring AI's filter expressions translate into SQL `WHERE` clauses on this JSONB column.

---

## HNSW Index

We use HNSW (Hierarchical Navigable Small World) — an index structure for approximate nearest-neighbor search.

Why HNSW over brute force?
- Brute force: O(n) comparisons for n vectors
- HNSW: O(log n) lookups using a graph structure
- HNSW trades a tiny amount of recall accuracy for much faster search

For this application with thousands of chunks, both work fine. For millions of vectors, HNSW becomes essential.

## Filtering by Document

A key requirement: when the user asks a question about document #5, we only want chunks from document #5.

Spring AI's filter expression system:

```java
FilterExpressionBuilder filterBuilder = new FilterExpressionBuilder();
var filter = filterBuilder.eq("documentId", String.valueOf(documentId)).build();

SearchRequest request = SearchRequest.builder()
    .query(question)
    .topK(5)
    .filterExpression(filter)
    .build();
```

This generates SQL like:
```sql
SELECT id, content, metadata
FROM vector_store
WHERE metadata->>'documentId' = '5'
ORDER BY embedding <=> $1  -- cosine distance to query embedding
LIMIT 5;
```

## Database Schema

```sql
-- Documents table (Hibernate-managed, UUID primary key)
CREATE TABLE documents (
    id                VARCHAR(36) PRIMARY KEY,  -- UUID generated by JPA
    name              VARCHAR NOT NULL,
    file_type         VARCHAR NOT NULL,
    stored_file_name  VARCHAR NOT NULL,
    file_size_bytes   BIGINT,
    uploaded_at       TIMESTAMP NOT NULL,
    status            VARCHAR NOT NULL,         -- UPLOADING | PROCESSING | READY | FAILED
    chunk_count       INTEGER
);

-- Chunks table (Hibernate-managed, IDENTITY Long PK)
CREATE TABLE document_chunks (
    id               BIGSERIAL PRIMARY KEY,
    document_id      VARCHAR(36) REFERENCES documents(id) ON DELETE CASCADE,
    content          TEXT NOT NULL,
    chunk_index      INTEGER NOT NULL,           -- 0-based position in document
    vector_store_id  VARCHAR UNIQUE              -- links to vector_store.id
);

-- Spring AI managed (1024-dim for jina-embeddings-v3):
CREATE TABLE vector_store (
    id        UUID PRIMARY KEY,
    content   TEXT,
    metadata  JSONB,
    embedding vector(1024)
);
```

Hibernate creates `documents` and `document_chunks` automatically via `ddl-auto=update`.
Spring AI creates `vector_store` via `initialize-schema=true`.
