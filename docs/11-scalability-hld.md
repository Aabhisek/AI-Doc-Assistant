# 11. Scalability вҖ” High-Level Design (HLD)

## Purpose

This document describes how the AI Document Assistant evolves from its current **single-node, single-user** deployment to a **production-grade, horizontally scalable** system. It answers:

- What breaks first as load increases?
- What does the scaled architecture look like at each tier?
- How do all the components change at the system level?
- What is the cloud-native target architecture?

---

## 9. Component Scaling Summary

| Component | Tier 0 (Current) | Tier 1 | Tier 2 | Tier 3 |
|-----------|-----------------|--------|--------|--------|
| **Frontend** | Docker (single) | S3 + CDN static | CDN + edge caching | Global CDN, multi-region |
| **Backend** | 1 Spring Boot instance | 1 instance + auth | 3вҖ“5 instances behind ALB | K8s Deployment, HPA |
| **Ingestion** | Synchronous, inline | Async (`@Async`) | Worker pool via RabbitMQ | K8s workers, HPA on queue depth |
| **Auth** | None | JWT (stateless) | JWT + refresh tokens | OAuth2 / OIDC (Keycloak) |
| **File Storage** | Local `./uploads/` | Amazon S3 | S3 + presigned URLs | S3 multi-region replication |
| **Relational DB** | Single PostgreSQL | RDS (managed) | RDS + 1 read replica | RDS Multi-AZ + 2 replicas |
| **Vector Store** | pgvector (embedded) | pgvector (RDS) | Qdrant (dedicated) | Qdrant cluster (distributed) |
| **Cache** | None | None | Redis (single) | Redis Cluster |
| **Message Queue** | None | RabbitMQ (single) | RabbitMQ (clustered) | Kafka (high throughput) |
| **LLM** | Groq free tier | Groq paid | GPT-4o or Groq Large | Multi-model routing |
| **Monitoring** | None | Actuator endpoints | Prometheus + Grafana | Full observability stack |
| **Search** | Pure vector | Pure vector | Pure vector | Hybrid BM25 + vector |
